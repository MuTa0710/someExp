# 驾驭智能

## Harness：驾驭智能的"马具"

Agent领域的名词 **Harness**，英文原意为马具。马具是人类为了驾驭马匹而设计的一套工具——如果LLM是"马匹"，那么Agent系统中，除开LLM以外的所有都可以算是Harness。我们设计Harness，就是设计各种约束方法，让LLM更好地完成任务。

这个类比背后，其实隐含着一套控制理论的逻辑。

## 控制理论的视角
[The Control Theory Behind Harness Engineering](https://adityashrishpuranik.com/writing/harness-engineering-as-control-theory)中提到：

> LLM（大语言模型）是被控对象，它产生输出。Harness是控制器，它负责约束、引导和校正。可观测性（日志、指标、测试、屏幕截图）是传感器。任务规范是参考信号。

从这个视角出发，使用智能体只有两种模式：

**开环控制**——你发出指令后就只能听天由命。目前大多数人使用智能体就是这么做的：输入任务，智能体运行，得到输出，然后祈祷一切顺利。

**闭环控制**——输出反馈到输入，系统能够自我纠正。这正是Harness工程的本质所在。也正因如此，尽管各个团队完全独立运作，但构建大规模实际系统的团队最终却采用了如此相似的架构。每一个成功案例，都是团队找到了闭环控制的方法；而每一个失败案例，都是团队采用了开环控制，然后纳闷为什么输出结果会偏离预期。

那么问题来了：既然闭环控制如此重要，为什么大多数团队在实践中仍然停留在开环控制的水平？我认为，一个深层原因是——我们长期被"数据决定论"所支配。

## 数据决定论的局限

周志华的《机器学习》讲到，模型是从数据中学到的知识。这个定义将数据和模型构造为了一个映射：在训练方法合理的情况下，模型的效果好不好，跟数据的好坏是强相关的。好数据练出来的模型，如果测试的数据也是"好"的，那结果就会好。这里的"好"可以理解为更贴近训练数据的分布，更贴近模型本身输出的分布。

"输入好，输出会好"——这个观念对我产生了很长一段时间的影响。对于模型测试结果的分析，我往往优先思考：训练数据能不能更好？提到数据对于深度学习的重要性时，我往往会说：这其实是一门数据科学。

直到我在使用langgraph构建Agent工作流时，这个假设开始动摇。

我构建了一个简单的问数Agent工作流，基于langgraph的图主导的流程，希望其能够实现用户通过自然语言对数据库中相关数据的查询，并且最终进行数据的总结。该工作流的实现很简单，接近于一个链表，所使用的模型参数也完全能够胜任该任务，甚至有点"大炮打蚊子"的意味。可是在运行测试时发现，简单的查询有时也会出错。

诚然，数据对于人工智能是基石一般的存在，模型的能力上限在预训练阶段结束后几乎就可以确定了。无论是后续的各阶段训练，都是"微调"，并不能让模型跨越其自身的智能上限。数据的质量着实对模型有着举足轻重的作用。

但是，当数据和模型能力都已经足够时，输出仍然不理想，瓶颈究竟在哪里？

## 瓶颈在Harness

如果我们从资源分配的角度去看待问题，会有一些新的发现。假设你的手上有一笔钱，将用于一个从零开始的AI Agent产品的研发投入。我们从最简单的模式出发，这笔钱会被投入到产品研发的三个部分：数据构造、模型训练、框架开发。我们当然希望最终产品能够在各种任务中取得好的效果，这是产品的目标。现在，如何分配资金才能让这个产品更好地贴近最终的目标？

数据构造和模型训练绝对会吃掉大部分的预算，用来保证Agent中最核心的LLM模型的能力。但是这笔钱是花得越多越好吗？[CL-bench: A Benchmark for Context Learning](https://arxiv.org/abs/2602.03587)也许给出了参考。文章通过精心设计的实验，系统性地证明了："输入是好的，输出也不会差"这一假设并不总是成立——即使输入质量很高，模型在复杂任务中的表现仍然可能出现显著退化。

这意味着，当数据和模型已经接近极致，输出不理想的根本原因很可能是：**我们缺乏一套有效的工程约束来组织、引导和纠正模型的推理过程。** 换言之，瓶颈不在"马匹"本身，而在"马具"。

## Harness的具体形态

模型的能力依然足够强大，但最终结果仍然表现不好。我们是否能从中获得启示？可以发现，产品研发时的资金投入也许有一点小小的问题——如果拉高框架开发的预算，结果会不会好一些？虽然本文仅供交流，无法给出严格的量化回答，但业界已经用行动投了票。

[learn-claude-code](https://github.com/shareAI-lab/learn-claude-code/blob/main/README-zh.md)这个仓库的README中有如下叙述：

> 模型负责思考。代码负责给模型提供工作环境。
> 这个"工作环境"就是 **Harness**。

对 coding agent 来说，harness 主要由这些部分组成：

- **Agent Loop**：不停地"向模型提问 -> 执行工具 -> 把结果喂回去"。
- **Tools**：读文件、写文件、改文件、跑命令、搜索内容。
- **Planning**：把大目标拆成小步骤，不让 agent 乱撞。
- **Context Management**：避免上下文越跑越脏、越跑越长。
- **Permissions**：危险操作先过安全关。
- **Hooks**：不改核心循环，也能扩展行为。
- **Memory**：把跨会话仍然有价值的信息保存下来。
- **Prompt Construction**：把系统说明、工具信息、约束和上下文组装好。
- **Tasks / Teams / Worktree / MCP**：让系统从单 agent 升级成更完整的工作平台。

以上是搭建一个Agent时需要考虑的"约束"。该仓库的作用如同字面意思，即教会读者从0到1开发一个类似Claude Code的Agent系统。从中也可以看出，对于当前AI Agent领域的头部团队，如OpenAI、Anthropic等，他们早已将Harness作为产品研发的重要战略。

回扣到控制理论的视角，这些组件并非零散的功能列表，而是一个完整的闭环控制系统：**Agent Loop** 构成了反馈回路，**Tools** 是执行器，**Planning** 和 **Permissions** 是控制器，**Context Management** 和 **Memory** 是状态观测器，**Prompt Construction** 是参考信号的编码器。正是这些组件的协同，让系统从"开环"走向了"闭环"。

## 总结

驾驭智能的关键，不在于让模型变得更聪明，而在于为模型设计一个足够好的工作环境——这就是Harness。

我们毫无疑问地一脚踏进了一个崭新的时代——这个时代由"过去几十年的数据积累"所孕育，诞生了对工作方式有着深远影响的工具。Harness作为Agent开发不可绕开的一个关键概念，给我们最大的启示是：无论何时，不要放弃用工程师的思维思考问题。面对LLM这个概率机器，人们是如何设计各种约束，让本有无数种路径的结局锚定在我们想要的地方——这才是驾驭智能的真正含义。
